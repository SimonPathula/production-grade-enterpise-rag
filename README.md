# Enterprise Agentic RAG

Enterprise Agentic RAG is a technical assistant for answering questions about enterprise IT documentation. The current domain focus is Kubernetes, Intel hardware, and enterprise networking. The app combines a FastAPI backend, a LangGraph agent workflow, Qdrant vector search, Hugging Face embeddings, Groq-hosted LLMs, and a Streamlit chat UI.

The goal is to give users grounded technical answers from uploaded documentation instead of relying only on the model's general knowledge. For off-topic prompts, the backend applies guardrails before running retrieval.

## What the App Does

The application has three main jobs:

1. Ingest technical documents.
   It reads files such as PDFs, HTML, text files, Word documents, and PowerPoint decks. The ingestion pipeline extracts text, chunks it, creates embeddings, stores processed artifacts in Google Cloud Storage, and indexes vectors in Qdrant.

2. Answer user questions with RAG.
   The backend classifies a user message as either conversational or technical. Technical questions go through document retrieval, reranking, and answer generation. Conversational messages are answered from chat history without retrieval.

3. Provide a Streamlit chat interface.
   The frontend is a Streamlit app under `ui/`. It sends user messages to the FastAPI backend and displays the answer, reasoning steps, and retrieved context chunks.

## Main Technologies

- Backend API: FastAPI
- Agent orchestration: LangGraph
- LLM provider: Groq through `langchain-groq`
- Main generation model: `llama-3.3-70b-versatile`
- Guardrail LLM when enabled: `llama-3.1-8b-instant`
- Embeddings: Hugging Face Inference API with `BAAI/bge-base-en-v1.5`
- Vector database: Qdrant
- Reranking: Groq LLM-based reranker
- Frontend: Streamlit
- Observability: Logfire and LangSmith
- Optional persistent memory: Postgres checkpointing through LangGraph
- Local fallback memory: LangGraph `MemorySaver`
- Cloud storage for ingestion artifacts: Google Cloud Storage

## How the Backend Works

The FastAPI backend lives in `app/main.py`. The main user endpoint is:

```text
POST /query
```

The request body is:

```json
{
  "query": "How do Kubernetes Jobs handle completions?",
  "thread_id": "user-session-id"
}
```

The response contains:

```json
{
  "question": "...",
  "answer": "...",
  "thought_process": ["Intent: Technical", "Retrieval: Required", "Context Retrieved"],
  "status": "Response generated.",
  "sources": ["retrieved chunk 1", "retrieved chunk 2"]
}
```

The backend flow is:

1. `guard()` checks whether the prompt should be blocked.
2. The LangGraph planner classifies the query as `CONVERSATIONAL` or `TECHNICAL`.
3. Technical queries are embedded with the Hugging Face API.
4. Qdrant returns candidate document chunks.
5. A Groq-based reranker chooses the most relevant chunks.
6. The responder sends the selected context and conversation history to the LLM.
7. The final answer, status, reasoning steps, and sources are returned to the UI.

## Guardrails

The project currently uses two guardrail modes.

Deterministic guardrails are always available. These block known off-topic prompts such as jokes, weather, movies, general trivia, and common jailbreak attempts before retrieval runs.

NeMo Guardrails support is still present in the code, but it is optional. It is controlled by:

```text
ENABLE_NEMO_GUARDRAILS=true
```

For Render deployment, NeMo Guardrails are disabled by default because the package can pull native `fastembed` and `onnxruntime` dependencies. Those dependencies caused deployment crashes on Render in this project. The Render requirements file therefore does not install `nemoguardrails`.

## Retrieval and Embeddings

The retrieval layer uses Qdrant and Hugging Face embeddings.

- Query embeddings are generated by `app/services/retrieval/embedding.py`.
- Qdrant search is handled by `app/services/retrieval/qdrant_service.py`.
- The Qdrant collection is currently set to `enterpriserag-hf`.
- Embedding model: `BAAI/bge-base-en-v1.5`.
- Vector size: 768 dimensions.

The backend requires a valid Hugging Face token because embeddings are generated through the hosted Hugging Face Inference API.

## Ingestion Pipeline

The ingestion code lives in `app/ingestion/processor.py`.

It can process:

- `.pdf`
- `.html` and `.htm`
- `.txt`
- `.docx`
- `.pptx`

The ingestion flow is:

1. Read files from a local directory.
2. Upload raw files to the configured raw GCS bucket.
3. Extract text using the correct loader.
4. Split text into chunks of about 800 characters with overlap.
5. Upload processed chunk JSON to the processed GCS bucket.
6. Embed chunks with Hugging Face.
7. Upsert vectors and payloads into Qdrant.

Example:

```bash
python -m app.ingestion.processor data
```

To wipe and recreate the Qdrant collection before ingesting:

```bash
python -m app.ingestion.processor data --wipe
```

## Streamlit Frontend

The Streamlit UI is in `ui/`.

The main deployed app should use:

```text
ui/streamlitapp.py
```

The frontend reads the backend URL from Streamlit Secrets or environment variables:

```toml
BACKEND_URL = "https://your-render-backend.onrender.com"
```

Do not include `/query` in `BACKEND_URL`. The app adds `/query` internally.

If `BACKEND_URL` is missing, the UI now shows a configuration error instead of silently calling `localhost:8000`.

## Required Environment Variables

The backend needs these values for normal RAG operation:

```text
GROQ_API_KEY=
QDRANT_CLUSTER_ENDPOINT=
QDRANT_API_KEY=
HF_API_TOKEN=
```

Optional but supported:

```text
LOGFIRE_TOKEN=
LANGSMITH_API_KEY=
LANGSMITH_TRACING=true
LANGSMITH_PROJECT=entrepriserag
LOCAL_MODE=true
ENABLE_NEMO_GUARDRAILS=false
```

For persistent Postgres memory:

```text
DB_HOST=
DB_USER=
DB_PASS=
DB_NAME=rag_memory
DB_CONNECTION_NAME=
LOCAL_MODE=false
```

For ingestion with Google Cloud:

```text
PROJECT_ID=
LOCATION=us-central1
GCP_RAW_BUCKET=
GCP_PROCESSED_BUCKET=
GCP_DOC_AI_LOCATION=
GCP_DOC_AI_PROCESSOR_ID=
```

## Running Locally

Create and activate a virtual environment, then install dependencies:

```bash
pip install -r requirements.txt
```

Start the backend:

```bash
uvicorn app.main:app --reload
```

Run the Streamlit UI:

```bash
streamlit run ui/streamlitapp.py
```

For local UI testing, set:

```text
BACKEND_URL=http://localhost:8000
```

## Deploying the Backend on Render

Use the Render-specific requirements file:

```bash
pip install -r requirements-render.txt
```

Start command:

```bash
uvicorn app.main:app --host 0.0.0.0 --port $PORT
```

Recommended Render environment variables:

```text
GROQ_API_KEY=
QDRANT_CLUSTER_ENDPOINT=
QDRANT_API_KEY=
HF_API_TOKEN=
ENABLE_NEMO_GUARDRAILS=false
LOCAL_MODE=true
```

After changing dependency files, use Render's "Clear build cache and deploy" once.

## Deploying the Frontend on Streamlit Community Cloud

Set the app entrypoint to:

```text
ui/streamlitapp.py
```

Set Streamlit Secrets:

```toml
BACKEND_URL = "https://your-render-backend.onrender.com"
LOGFIRE_TOKEN = "optional"
```

Then reboot the Streamlit app. The sidebar shows the backend URL currently being used.

## Current Deployment Notes

- Render backend should use `requirements-render.txt`.
- Streamlit frontend must define `BACKEND_URL` in Streamlit Secrets.
- NeMo Guardrails are optional and disabled by default for Render.
- Qdrant and Hugging Face are required for technical RAG queries.
- Groq is required for planning, reranking, and answer generation.
- If `LOCAL_MODE=true`, LangGraph uses in-memory checkpoints.
- If `LOCAL_MODE=false`, the app tries to use Postgres checkpointing and falls back to memory if the connection fails.

## Tests

Run the focused guardrail tests:

```bash
python -m pytest tests/test_guardrails.py -q
```

The repository also includes evaluation scripts under `tests/` for routing, retrieval, context latency, and RAG answer quality.

## Known Limitations

- Render deployment currently uses deterministic guardrails only unless NeMo is explicitly enabled and compatible native dependencies are available.
- The app depends on hosted Hugging Face embeddings, so technical retrieval will fail if the Hugging Face token is missing or invalid.
- The RAG quality depends on the Qdrant collection being populated with the expected 768-dimensional embeddings.
- Streamlit and FastAPI are deployed separately, so the frontend must point to the correct backend URL.
